# Team Handbook

Version: 1.0

---

# 1. Team Purpose

Our goal is not only to build a working project but also to learn how to collaborate effectively as an engineering team.

Success means:

* Delivering a functional MVP by the deadline.
* Maintaining a clear and organized development process.
* Ensuring every team member contributes.
* Learning from the project experience.

---

# 2. Team Principles

## Clarity Over Assumptions

If something is unclear, ask.

Do not make assumptions about requirements, architecture, or implementation details.

---

## Small Steps Over Large Tasks

Large tasks are difficult to estimate and review.

Break work into small tasks that can be completed within a few hours.

---

## Working Product Over Perfect Product

A simple working MVP is better than a perfect unfinished system.

Build first. Improve later.

---

## Transparency Over Silence

If you encounter a blocker or problem, communicate it.

A hidden problem eventually becomes a team problem.

---

## No Hero Rule

The project must never depend on a single person.

Every critical part of the system should be understood by at least two team members.

If someone creates an important component, they should:

* Explain how it works.
* Document important decisions.
* Ensure the team understands the overall design.

Our goal is to build a sustainable project, not create an irreplaceable developer.

---

# 3. Team Structure

## Team Lead

Responsibilities:

* Define project direction.
* Maintain project specifications.
* Prioritize tasks.
* Review pull requests.
* Coordinate the team.
* Make final decisions when consensus cannot be reached.

The Team Lead is responsible for project success, not for doing all the work.

---

## Team Members

Responsibilities:

* Complete assigned tasks.
* Ask questions when blocked.
* Keep GitHub up to date.
* Review their own work before opening a Pull Request.
* Communicate progress honestly.

---

# 4. Decision Making

We use a hybrid decision-making model.

## Discussion

Everyone is encouraged to propose ideas and solutions.

Technical discussions should remain open and constructive.

---

## Final Decision

If consensus is reached:

Decision = Team Agreement

If consensus is not reached:

Decision = Team Lead

This prevents endless debates and keeps progress moving.

---

# 5. Communication

Primary communication platform:

Telegram

---

## Response Expectations

During active development:

* Respond within 24 hours.
* Inform the team if you will be unavailable.

---

## Asking for Help

Do not spend multiple days stuck on the same problem.

If you cannot solve a blocker within one hour:

1. Investigate independently.
2. Document the issue.
3. Ask for help.

---

# 6. Meetings

## Offline Meetings

Frequency:

1–2 times per week

---

Agenda:

### Progress Review

What has been completed?

### Current Problems

What is blocking progress?

### Next Goals

What should be completed before the next meeting?

---

Duration:

60–90 minutes

---

# 7. Development Workflow

GitHub is the source of truth for the project.

If information in Telegram and GitHub conflicts, GitHub takes priority.

---

## GitHub Issues

Every task must exist as an Issue.

Examples:

* Create upload API
* Implement ZIP parser
* Build homepage
* Design database schema

---

## Issue Requirements

Each Issue should contain:

* Goal
* Description
* Expected Result

If a task is unclear, clarify it before starting work.

---

## Labels

Recommended labels:

* frontend
* backend
* ai
* bug
* documentation
* enhancement
* good-first-issue

---

# 8. Branch Strategy

Never work directly on the main branch.

Branch naming:

feature-issue-number-short-name

Examples:

* feature-12-upload-api
* feature-18-home-page
* bugfix-21-parser-error

---

# 9. Pull Requests

Before opening a Pull Request:

* Code compiles.
* Code runs correctly.
* Related Issue exists.
* Basic testing has been completed.

---

Pull Request Template:

### What was implemented?

### Related Issue

### How was it tested?

### Additional Notes

---

Every Pull Request must receive at least one review before merging.

# 11. Definition of Done

A task is considered complete only when:

* Functionality is implemented.
* Code works correctly.
* Pull Request is merged.
* No known critical issues remain.
* Related Issue is closed.

If any item is missing, the task is not Done.

# 13. Leadership Principle

Reduce uncertainty.

Team members create progress.

Together, the team creates results.

1. Create an Issue.
2. Discuss the problem.
3. Make a decision.
